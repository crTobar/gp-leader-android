## 1. SQL aplicado en Supabase

```sql
ALTER TABLE member_activity_record
  ADD COLUMN IF NOT EXISTS status text NOT NULL DEFAULT 'draft'
  CHECK (status IN ('draft', 'approved', 'pending_board', 'rejected'));

CREATE INDEX IF NOT EXISTS idx_mar_status ON member_activity_record(status);
```

### Estados del campo `status`
| Estado | Significado |
|---|---|
| `draft` | Recién enviado por miembro, esperando aprobación del líder |
| `approved` | Aprobado por líder (actividades no monetarias) |
| `pending_board` | Aprobado por líder pero requiere validación de Junta de Iglesia (monetarias) |
| `rejected` | Rechazado por líder |

## 3. `ActividadRepositoryImpl.kt` — cambios

### Actividades de miembros nacen en `draft`
`toggleMiembroActividad` y `upsertContadorSemanalMiembro` ahora incluyen `put("status", "draft")` en el upsert, de modo que toda actividad auto-registrada por un miembro empieza en borrador hasta que el líder la apruebe.

### Totales solo suman lo aprobado
`getActividadesConTotales` ahora filtra `member_activity_record` por `status IN ('approved', 'pending_board')` — los drafts no se suman al acumulado visible.

### Implementación de los 4 métodos nuevos
- `getPendingMemberActivities`: obtiene borradores de un tipo de actividad para el grupo
- `getPendingCountPerTipo`: devuelve un `Map<tipoId, count>` con cuántos borradores hay por actividad
- `approveMemberActivity`: cambia status a `approved` o `pending_board` (según si es monetaria); permite corregir count/monto
- `rejectMemberActivity`: cambia status a `rejected`

## 4. `RegistroViewModel.kt` — lógica de desglose invertida

### `tieneDesglose` ampliado
```kotlin
// ANTES: solo PASTOR+CONTADOR y PARTICIPANTES
val tieneDesglose = (nivel == NivelActividad.PASTOR && tipo == TipoMarcador.CONTADOR) ||
                    tipo == TipoMarcador.PARTICIPANTES
```

### `desgloseExpandido = true` por defecto
Las actividades se crean con `desgloseExpandido = true` para que el detalle por miembro esté visible al entrar a `DetalleActividadScreen`.

### Auto-cálculo del total desde el desglose (dirección invertida)
| Función | Antes | Ahora |
|---|---|---|
| `onDesgloseChange` | Limitaba cada miembro al `totalGeneral - sumOtros` | Suma libremente; `cantidad = sumOf { it.cantidad }` |
| `onDesgloseMontoChange` | Limitaba por `montoTotal - sumOtros` | Suma libremente; `monto = sumOf { it.montoDesglose }` |
| `onDesgloseParticipacionChange` | Limitaba por `totalGlobal` | `cantidad = count { it.participo }` |

## 5. `RegistroPaso2Screen.kt` — NeuCards individuales por actividad

### Estructura anterior
Un solo `NeuCard` envolvía el header + todas las `ActividadRow` con controles inline (ContadorInline, MontoInline, desglose expandible).

### Estructura nueva
Cada actividad tiene su propia `NeuCard` neumórfica:
- Ícono de tipo (₡, contador, P)
- Nombre + unidad + acumulado del periodo
- **Chip de valor**: `—` cuando vacío, o el total calculado en `BackgroundDeep`
- **CHECKBOX**: toggle inline dentro de su NeuCard
- **Editables**: chevron `→` en Accent; **Bloqueadas**: candado `🔒` en Muted
- Toda edición de contadores/montos ocurre en `DetalleActividadScreen`

### `SeccionActividades` simplificada
- Header: etiqueta de nivel como pill con color (Ink/gris oscuro/BackgroundDeep)
- Cards individuales con `Arrangement.spacedBy(8.dp)`

### Parámetros eliminados de `RegistroPaso2Content`
`onCantidadChange`, `onMontoChange`, `onToggleDesglose`, `onDesgloseChange`, `onDesgloseMontoChange`, `onDesgloseParticipacionChange` — ya no se pasan desde Paso 2; solo viven en `DetalleActividadScreen`.

## 6. `DetalleActividadScreen.kt` — entrada granular por miembro

### Cambios principales
| Elemento | Antes | Ahora |
|---|---|---|
| Total | `ContadorGrande` (editable por el usuario) | `TotalReadOnly` (calculado automáticamente del desglose) |
| Desglose | Colapsable, solo CONTADOR | Siempre expandido, soporta CONTADOR + MONETARIO + PARTICIPANTES |
| Límite por miembro | Limitado a `totalGeneral - sumOtros` | Sin límite (`sinLimite = true`) — cada miembro ingresa su propio valor |
| `onGuardar` | Guardaba `cantidad` + `notas` | Solo guarda `notas` (el total vive en el VM vía desglose) |
| `var cantidad` | Estado local | Eliminado — se lee de `actividad.cantidad` (estado del VM) |
| `var desgloseExpandido` | Estado local iniciando en `false` | Eliminado — siempre expandido |

### Nuevo composable `TotalReadOnly`
Muestra el total calculado con `displayLarge` + sublabel de unidad. Formato según tipo:
- MONETARIO: `₡{monto.toLong()}`
- PARTICIPANTES: `{count} participantes`
- Resto: `{cantidad} {unidad}`
- Vacío: `—` en Muted

### Entry point ampliado
Ahora conecta los 3 handlers de desglose al ViewModel:
```kotlin
onDesgloseChange              = { miembroId, cant   -> viewModel.onDesgloseChange(...) }
onDesgloseMontoChange         = { miembroId, monto  -> viewModel.onDesgloseMontoChange(...) }
onDesgloseParticipacionChange = { miembroId, checked -> viewModel.onDesgloseParticipacionChange(...) }
```

## 8. `bible_study` — nueva tabla Supabase

```sql
CREATE TABLE IF NOT EXISTS bible_study (
    id               UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    member_id        UUID        NOT NULL REFERENCES member(id) ON DELETE CASCADE,
    student_name     TEXT        NOT NULL,
    completed_lessons INTEGER[]  NOT NULL DEFAULT '{}',
    created_at       TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_bible_study_member_id ON bible_study(member_id);
```

- `completed_lessons`: array de enteros 1–20 con las lecciones completadas (ej: `{1,2,3}`)
- `currentLesson` calculado client-side: `(max(completed_lessons) + 1).coerceAtMost(20)`
